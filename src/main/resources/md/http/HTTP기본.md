# HTTP

### 1. 모든 것이 HTTP(HyperText Transfer Protocol)
> - 문서간의 링크를 통해서 연결할 수 있는 html 프로토콜로 시작했지만
> - 지금은 모든 것을 전송!
>   - HTML, TEXT
>   - IMAGE, 음성, 영상, 파일
>   - JSON, XML(API
>   - 거의 모든형태의 데이터 전송 가능
>   - 서버간에 데이터를 주고 받을 때도 대부분 HTTP 사용
>   - 지금은 HTTP 시대!
>   - TCP를 직접 연결해서 사용하는 경우는 게임서버나 특수한경우에만 하게 될 것임
>   - 요즘 모바일 게임도 HTTP로 열어서 통신하는 구조로 개발 많이 함

### 2. HTTP 역사
> - HTTP/0.9 1991년 : GET 메서드만 지원, HTTP 헤더X
> - HTTP/1.0 1996년 : 메서드, 헤더 추가
> - HTTP/1.1 1997년 : 가장 많이 사용, 우리에게 가장 중요한 버전
>   - RFC2068(1997) -> RFC2616(1999) -> RFC7230~7235(2014)
> - HTTP/2 2015년 : 성능 개선
> - HTTP/3 진행중 : TCP 대신에 UDP사용, 성능 개선

### 3. 기반 프로토콜
> - TCP : HTTP/1.1, HTTP/2 
> - UDP : HTTP/3
> - 현재 HTTP/1.1 주로 사용
>   - HTTP/2, HTTP/3도 점점 증가
> - HTTP/1.1 만 배워두면 그 후 버전은 성능개선!

> - HTTP 메시지
> - 단순함, 확장 가능

### 5. 클라이언트 서버 구조
> - Request Response 구조
> - 클라이언트는 서버에 요청을 보내고, 응답을 대기
> - 서버가 요청에 대한 결과를 만들어서 응답
> - 분리하는 것이 중요
>   - 클라이언트와 서버 분리
>     - 예전에는 클라이언트와 서버가 분리되어있지 않았음
>     - 클라이언트 서버 개념적으로 분리
>     - 비지니스 로직, 데이터 서버에 모아둠
>     - 클라이언트는 UI, 사용성에 집중
>     - 클라이언트와 서버가 각각 독립적으로 진화 할 수 있음
>     - 클라이언트는 UI, UX에 집중!
>     - 서버는 트래픽이 막 100배 폭주했을 경우 아키텍처를 어떤식으로 할지, 백엔드 기술을 어떻게 대용량 트래픽으로 더 고도화 진화할 지만 고민
>       - php를 사용하다 java로 변경

### 6. 무상태 프로토콜(Stateless)
> - 서버가 클라이언트의 상태를 보존X
> - 장점 : 서버 확장성 높음(스케일 아웃)
> - 단점 : 클라이언트가 추가 데이터 전송
> - 상태 유지(Stateful) : 중간에 다른 점원으로 바뀌면 안됨
>   - 항상 같은 서버가 유지되어야 함
>   - 서버 1번이 장애가 나면 클라이언트에서 부터 다시 요청을 시작해야 함
>     - 중간에 다른 점원으로 바뀔 때 상태 정보를 다른 점원에게 미리 알려주어야 함
> - 무상태(Stateless) : 중간에 다 른 점원으로 바뀌어도 됨
>   - 클라이언트가 애초에 데이터를 다 담아서 보냄
>   - 아무 서버나 호출해도 됨
>   - 중간에 서버 1번이 장애가 나면 중계 서버가 2번으로 던지면 됨
>     - 갑자기 고객이 증가해도 점원을 대거 투입할 수 있음
>     - 갑자기 클라이언트 요청이 증가해도 서버를 대거 투입할 수 있음
>   - 스케일 아웃 - 수평 확장 유리
>   - 백엔드 장비를 수십대 수백대로 한번에 늘릴 수 있음
> - 무상태는 응답 서버를 쉽게 바꿀 수 있다. -> 무한한 서버 증설 가능
>
> - Stateless 실무 한계
>   - 모든 것을 무상태로 설계 할 수 있는 경우도 있고 없는 경우도 있다.
>   - 무상태
>     - 예) 로그인이 필요 없는 단순한 서비스 소개 화면
>   - 상태 유지
>     - 예) 로그인
>   - 로그인한 사용자의 경우 로그인 했다는 상태를 서버에 유지
>   - 일반적으로 브라우저 쿠키와 서버 세션등을 사용해서 상태 유지
>   - 상태 유지는 최소한만 사용
>   - 데이터를 너무 많이 보내야 하는 단점
> 
> 웹 애플리케이션을 설계할 때 최대한 무상태로 설계!


### 7. 비 연결성(connectionless)
> - 연결을 유지하는 모델(TCP/IP)
>   - 클라이언트1 요청 -> 연결 유지 <- 서버 응답
>   - 클라이언트2 요청 -> 클라이언트1 접속되어있는 상태 <- 서버 응답
>   - 클라이언트3 요청 -> 클라이언트1, 2 접속되어있는 상태 <- 서버 응답
>   - 클라이언트2, 3이 놀고 있어도 계속 연결을 서버가 유지해야하는 단점
> 
> - 연결을 유지하지 않는 모델(TCP/IP)
>   - 클라이언트1 요청 -> 서버 응답 -> 연결 끊음
>   - 클라이언트2 요청 -> 서버 응답 -> 연결 끊음
>   - 클라이언트3 요청 -> 서버 응답 -> 연결 끊음
>   - 자원을 현재 요청을 주고받을 때만 연결을 하고 그 다음 끊어서 서버가 유지하는 자원을 최소한으로 유지
>
> - HTTP는 기본이 연결을 유지하지 않는 모델
> - 일반적으로 초 단위의 이하의 빠른 속도로 응답
> - 1시간 동안 수천명이 서비르를 사용해도 실제 서버에서 동시에 처리하는 요청은 수십개 이하로 매우 작음
>   - 예) 웹 브라우저에서 계속 연속해서 검색 버튼을 누르지는 않는다.
> - 서버 자원을 매우 효율적으로 사용할 수 있음
> - HTML문서를 서로 주고 받는 환경에서는 연결이 유지되지 않는 것이 서버입장에서 더 자원의 가용성을 높일 수 있음
> 
> - 단점
>   - TCP/IP 연결을 새로 맺어야 함 - 3 way handshake 시간 추가
>   - 웹 브라우저로 사이트를 요청하면 HTML뿐만 아니라 자바스크립트, css, 추가 이미지 등등 수 많은 자원이 함께 다운로드
>   - 지금은 HTTP 지속연결(Persistent Connections)로 문제 해결
>   - HTTP/2, HTTP/3에서 더 많은 최적화
>
> - HTTP 초기 - 연결, 종료 낭비
>   - 연결 -> 요청/HTML 응답 -> 종료
>   - 연결 -> 요청/자바스크립트 응답 -> 종료
>   - 연결 요청/이미지 응답 -> 종료
> 
> - HTTP지속연결(Persistent Connections)
>   - 연결 -> 요청/HTML 응답, 요청/자바스크립트 응답, 요청/이미지 응답 -> 종료
> 
> - HTTP/2, HTTP/3는 이런 것들이 더 많이 빠르게 해결 됨
> - HTTP/3는 UDP프로토콜이라는 걸 사용하면서 연결 속도 자체도 확 줄임
>
